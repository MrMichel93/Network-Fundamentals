Thank you for your interest in contributing to this course! We welcome improvements, bug fixes, and new content.

## How to Contribute

### Reporting Issues

Found a bug or error? Have a suggestion? Please open an issue!

**Before opening an issue:**
- Search existing issues to avoid duplicates
- Check if it's already fixed in the latest version

**When opening an issue:**
- Use a clear, descriptive title
- Describe the problem or suggestion in detail
- Include steps to reproduce (for bugs)
- Mention which module/lesson is affected

### Suggesting Enhancements

We love ideas for improving the course! When suggesting enhancements:
- Explain why this would be useful
- Provide examples if possible
- Consider whether it fits the beginner-to-intermediate level

### Contributing Code or Content

#### Types of Contributions

1. **Fix typos or errors** in lessons
2. **Improve code examples** (add comments, fix bugs, optimize)
3. **Add exercises or projects**
4. **Create diagrams or visualizations**
5. **Improve documentation**
6. **Add translations** (future)

#### Getting Started

1. **Fork** this repository
2. **Clone** your fork locally
3. **Create a branch** for your changes
   ```bash
   git checkout -b fix/typo-in-http-lesson
   # or
   git checkout -b feature/new-exercise
   ```
4. **Make your changes**
5. **Test** your changes (run code examples, check formatting)
6. **Commit** with a clear message
   ```bash
   git commit -m "Fix typo in HTTP Fundamentals lesson"
   ```
7. **Push** to your fork
8. **Open a Pull Request**

#### Code Style

**Python:**
- Follow PEP 8
- Use meaningful variable names
- Add comments for complex logic
- Include docstrings for functions

**Markdown:**
- Use proper heading hierarchy (# ## ###)
- Include code fences with language specification
- Add blank lines between sections
- Use emoji sparingly and consistently

**Code Examples:**
- Must be runnable
- Include dependencies in comments
- Add clear comments
- Keep examples simple and focused

#### Writing Style

- **Be clear and concise**
- **Use analogies** for complex concepts
- **Be encouraging** - remember the audience
- **Use examples** from real-world scenarios
- **Progressive difficulty** - start simple, build up
- **Active voice** preferred over passive
- **Second person** ("you" instead of "the student")

### Pull Request Process

1. **Describe your changes** in the PR description
2. **Link related issues** if applicable
3. **Wait for review** - we'll review as soon as possible
4. **Address feedback** if reviewers request changes
5. **Celebrate** when your PR is merged! üéâ

### Commit Message Guidelines

Use clear, descriptive commit messages:

```
Fix typo in HTTP status codes section

Changed "200 Sucess" to "200 Success" in 02-HTTP-Fundamentals/README.md
```

**Good commit messages:**
- `Add WebSocket chat example`
- `Fix broken link in Prerequisites`
- `Improve REST API design explanation`
- `Add exercise solution for Module 3`

**Avoid:**
- `Update`
- `Fix stuff`
- `WIP`
